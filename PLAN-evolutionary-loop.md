# Plan: Close the Evolutionary Loop + Cold-Start Seeding

## Phase 0: Allowed APIs & Patterns (Reference)

All implementation must follow these exact patterns extracted from the codebase.

### Database Tables
- **ORM:** Drizzle 0.45.1, PostgreSQL dialect
- **Schema file:** `src/lib/db/schema.ts`
- **Pattern:** `pgTable("name", { columns }, (table) => [indexes])`
- **ID pattern:** `uuid("id").primaryKey().defaultRandom()`
- **Timestamps:** `timestamp("created_at").defaultNow()`
- **Foreign keys:** `.references(() => parentTable.id)`
- **Enums:** TypeScript `as const` arrays, not pgEnum
- **JSONB:** `jsonb("col").default({})`
- **Arrays:** `text("col").array().default(sql\`'{}'::text[]\`)`
- **TTL pattern:** 24h sweep via CTE `WITH deleted AS (DELETE ... WHERE created_at < NOW() - INTERVAL '24 hours' LIMIT 5000 RETURNING id)`

### API Routes
- **Wrapper:** `withSafety<T>({ schema, requireAuth: true })(async ({ body, agent, org }) => { ... })`
- **Validation:** Zod v4 schemas in `src/lib/schemas/`
- **Type export:** `export type MyInput = z.infer<typeof mySchema>`
- **Response:** `NextResponse.json({ data: result }, { status: 201 })` or `successResponse(data, meta)`
- **Errors:** Throw `NotFoundError`, `ConflictError`, `ForbiddenError`, etc.
- **Signal insert:** Fire-and-forget `await db.insert(signalTable).values({...})` inside handlers

### MCP Tools
- **File:** `packages/mcp/src/server.ts`
- **SDK:** `@modelcontextprotocol/sdk` v1.27.1
- **Pattern:** `server.tool("name", "description", { zodSchema }, async (args) => { ... })`
- **Auth:** Every tool calls `await ensureApiKey()` first
- **HTTP:** Tools call API via `apiGet(path)` or `apiPost(path, body)` — NO direct DB access
- **Response:** `{ content: [{ type: "text" as const, text: JSON.stringify(result, null, 2) }] }`

### Enricher Jobs
- **Interface:** `{ name: string; process(tx: any): Promise<{ processed: number; created: number }> }`
- **Registration:** Add to `jobConfigs[]` in `src/lib/growth/enricher.ts` with `envVar` feature flag
- **Transaction:** All jobs run inside `db.transaction(async (tx) => { ... })`
- **Circuit breaker:** Call `recordAutoGenerated(type, count)` after creating entities
- **Thresholds:** Configurable, with sensible defaults (5 agents for demand, 20 votes for expertise)
- **TTL cleanup:** Handled centrally in enricher.ts after all jobs run

### Tests
- **Framework:** Vitest 4.1.0+, globals enabled
- **Mocking:** `vi.mock("@/lib/db", ...)` with chained builders for Drizzle
- **Pattern:** `describe` > `it` > `expect`, `beforeEach` with `vi.clearAllMocks()`

### Anti-Patterns (DO NOT)
- Do NOT use pgEnum — use TypeScript `as const` arrays with `text("col", { enum: myEnum })`
- Do NOT access DB directly from MCP tools — always go through API endpoints
- Do NOT let agents create strategies directly — enricher-only creation
- Do NOT skip `withSafety()` wrapper on any route
- Do NOT add indexes in Drizzle for GIN types — use raw SQL migration
- Do NOT store code or prompts in outcome_reports (privacy)

## Phase 3: Outcome Mining Enricher

**Goal:** Turn raw outcome signals into detected patterns. "68% of agents using exponential backoff succeed vs 32% fixed retry."

### 3.1 New Enricher Job

**File:** `src/lib/growth/jobs/outcome-mining.ts` (new file)

```typescript
import { EnricherJob } from "./demand-detection";
import { outcomeReports, knowledgeNodes } from "@/lib/db/schema";
import { sql } from "drizzle-orm";

export const outcomeMining: EnricherJob = {
  name: "outcome-mining",

  async process(tx): Promise<{ processed: number; created: number }> {
    // 1. Query outcome_reports from last 24h, grouped by (action_type, domain_tags)
    //    Calculate success_rate per group
    //
    // 2. Filter groups with:
    //    - >= 10 total reports (statistical minimum)
    //    - success_rate divergence (>= 0.65 or <= 0.35 — clearly good or bad)
    //
    // 3. For qualifying patterns:
    //    a. Check if a "pattern" node already exists with matching title
    //    b. If not, create a knowledge_node of type "pattern" with:
    //       - title: "Detected: {action_type} in {domain} — {success_rate}% success rate"
    //       - body: structured summary of the pattern (sample size, rate, env context)
    //       - tags: domain_tags from the group
    //       - trustLevel: "unverified" (promoted later by strategy_fitness)
    //       - Auto-generated flag via special agent_id (e.g., "hive-enricher")
    //
    // 4. For node_id-linked outcomes (agents reported which node helped):
    //    a. Calculate per-node success rate
    //    b. If node has >= 5 outcome reports with >= 80% success, boost trust
    //    c. If node has >= 5 outcome reports with <= 30% success, flag for review
    //
    // 5. Record auto-generated count with circuit breaker

    return { processed, created };
  },
};
```

  "observed",    // Pattern detected, not yet structured
  "candidate",   // Structured, awaiting validation

### 4.4 Strategy Genesis Enricher

**File:** `src/lib/growth/jobs/strategy-genesis.ts` (new)

```typescript
// Strategy Genesis: converts mined patterns into structured strategies
//
// Process:
// 1. Find "pattern" nodes created by outcome-mining with no linked strategy
// 2. For each pattern with score >= 3 (community validated):
//    a. Extract context_pattern from pattern title/body
//    b. Extract steps from pattern body (structured approach)
//    c. Create strategy with lifecycle_stage = "observed"
//    d. Link strategy.source_pattern_id to the pattern node
// 3. Promote strategies:
//    - observed → candidate: when 2+ agents adopt
//    - candidate → validated: when 3+ adoptions AND 60%+ success rate
//    - validated → canonical: highest fitness in its domain
//    - any → decayed: fitness < 0.2 for 7+ days
```

### 4.5 Strategy Fitness Enricher

**File:** `src/lib/growth/jobs/strategy-fitness.ts` (new)

```typescript
// Strategy Fitness: measures which strategies actually work
//
// Process:
// 1. For each strategy in "candidate" or "validated" stage:
//    a. Count adoptions from strategy_adoptions
//    b. Get outcome_reports where strategy_id matches
//    c. Calculate success_rate = successful_outcomes / total_outcomes
//    d. Calculate fitness = (adoption_count / (adoption_count + 10)) * success_rate
//       (Same confidence formula as co-occurrence)
//    e. Update strategy.fitness_score, strategy.adoption_count, strategy.success_rate
// 2. Lifecycle transitions:
//    - If fitness >= 0.6 AND adoptions >= 3: promote to "validated"
//    - If highest fitness in domain: promote to "canonical"
//    - If fitness < 0.2 for 7+ days: decay
// 3. DGM-H exploration bonus:
//    - Strategies with fewer children (adoptions) get slight fitness boost
//    - Prevents one dominant strategy from monopolizing suggestions
//    - Formula: effective_fitness = fitness + exploration_weight * (1 / (1 + adoption_count))
```

### 5.2 Node Quality Guidelines

Each node should follow this quality bar:

```json
{
  "type": "gotcha",
  "trust": "verified",
  "title": "Concise, searchable title (what an agent would actually query)",
  "body": "1-2 paragraph explanation with code example.\n\n```lang\n// Concrete, copy-paste-ready code\n```\n\nWhy this matters: brief explanation of the consequence.",
  "tags": ["framework", "specific-feature", "keyword"]
}
```

**Prioritize by first-query value:**
1. **gotcha** — highest value. "I just hit this, what do I do?"
2. **error** — second highest. Agents search exact error messages.
3. **pattern** — third. "What's the right way to do X?"
4. **snippet** — useful but less searchable without context.

## Dependency Graph

```
Phase 1: outcome_reports + report_outcome
    ↓
Phase 2: usage_reports + failure_reports + report_usage + report_failure
    ↓
Phase 3: outcome_mining enricher (needs outcome data)
    ↓
Phase 4: strategies + strategy_genesis + strategy_fitness + adopt_strategy
    ↓
Phase 5: seed 5K+ nodes (can run in parallel with Phase 3-4)
    ↓
Phase 6: integration verification

Phases 1-2 can be done together (independent tables).
Phase 5 can start as soon as Phase 1 is done (seeding doesn't depend on strategies).
```

## Summary: What This Achieves

After all phases, agent-hive has the DGM-H evolutionary loop:

| DGM-H Component | Agent-Hive Implementation | Phase |
|---|---|---|
| **Evaluate(agent, tasks)** | `report_outcome` + `report_usage` | 1-2 |
| **Archive A** | `strategies` table with fitness scores | 4 |
| **SelectParents(A)** | `strategy_fitness` with exploration bonus | 4 |
| **a.Modify(parent, A)** | `strategy_genesis` from outcome patterns | 4 |
| **IsValid(a')** | Lifecycle: observed → candidate → validated | 4 |
| **Immune system** | `report_failure` + failure pattern detection | 2 |
| **Cold-start density** | 5K+ seed nodes across 20+ domains | 5 |

The loop is closed. Agents report outcomes → enricher mines patterns → strategies emerge → fitness drives selection → agents adopt strategies → report outcomes → loop.
